### 2026-04-17 — Voice input "nothing at all" on Pixel 7 Pro + first-words drop

**Error:** Two symptoms at once — (1) first 3-5 s of audio missing for several days, (2) today nothing transcribed at all on Pixel 7 Pro.

**Root Cause (12 overlapping issues, documented as Causes A–L in the plan):**
- **F (today's primary)**: `new AudioContext({ sampleRate: 16000 })` is silently ignored on Android Chrome — context runs at 48 kHz native rate. Worklet emitted 1600 samples of 48 kHz data labelled `audio/pcm;rate=16000`; Gemini interpreted it as 16 kHz → audio plays ~3× too fast → empty transcript. No client-side error.
- **A (first-3-5-seconds drop)**: `self._active = True` was set only after the Gemini token fetch (~1 s) + WS handshake (~0.5 s) completed. Audio chunks arriving during boot hit the `if not self._active: return` guard and were silently dropped.
- **B**: `_flush_loop` did `activity_end → sleep(50ms) → activity_start`. Audio sent during that gap was discarded server-side (disabled-VAD mode).
- **C**: `_stop_voice` didn't wait for the tail transcription — last 200-500 ms of audio's transcription never arrived.
- **D**: `_start_voice` was an untrapped `loop.create_task` — any failure (bad token, rotated model, WS fail) became a logged task error the PWA never heard about; PWA stayed stuck on "listening".
- **E**: `MODEL = "gemini-2.5-flash-native-audio-latest"` was a floating alias. Google rotates these silently.

**Fix (shipped 2026-04-17):** 12-step hardening pass, one commit. See [memory/project_audio_pipeline.md](C:/Users/Korisnik/.claude/projects/c--Antigravity-Voicer/memory/project_audio_pipeline.md) for the full map. Highlights:
- Worklet reads `processorOptions.inputSampleRate`, anti-aliases with 7-tap windowed-sinc FIR low-pass at ~7.5 kHz, decimates to 16 kHz with fractional-sample `_phase` carryover for non-integer ratios (44.1k→16k). Bypass path at 16 kHz for iOS Safari.
- `_active = True` flipped as first line of `GeminiLive.start()`, before any await.
- `asyncio.Lock` brackets `activity_end/start` pair in `_flush_loop` and `send_realtime_input` in `_send_loop`. Sleep removed.
- `flush_final(timeout=1.5)` drains queue, waits on an `asyncio.Event` signaled by `_recv_loop` on next `turn_complete` or `input_transcription`. Short-circuits if `_turn_active` already False (no deadlock).
- Error taxonomy: `token` / `model` / `handshake` / `no-audio` / `unknown`. `_send_status` queues to `_pending_status_flushes` when channel not open. PWA shows `friendlyMessageFor(reason)` copy.
- `MODEL = "gemini-2.5-flash-preview-native-audio-dialog"` pinned explicitly; env override available; `ALLOWED_MODELS` allowlist enforced in test.
- 3 s startup + 10 s mid-session audio watchdogs. Track-mute rebuild. start/stop race via `_voice_starting_task`. Rolling-5s RMS warning at < 0.0002 (accounting for `noiseSuppression: true`).
- Test coverage: 31 host unit tests + 4 PWA sample-rate + 4 rapid-toggle + 11 error-status + 1 real-API integration (parameterized BYOK/hosted, opt-in via `RUN_REAL_API_TESTS=1`). Extracted `WORKLET_CODE` + `friendlyMessageFor` to `src/hooks/voice-worklet.ts` so `.mjs` tests can import without React.

**Key Rule:** NEVER trust `AudioContext.sampleRate` to match what was requested — always read it back and route through the decimation worklet if it doesn't match 16 kHz. NEVER set an `_active` flag after any `await` in a session-start path. The `flush_final()` path is the only supported voice-stop — do not bypass it with inline `interim_buffer.strip()` reads.

### 2026-03-26 — Trackpad mode taps landing at wrong position

**Error:** In trackpad/mouse mode, tapping on the phone screen clicked at the wrong position on the PC — off by a wide margin.

**Root Cause:** Trackpad taps sent `{ type: "click" }` which fires `pyautogui.click()` at the invisible cursor's current position (wherever relative mouse movements accumulated to). But the user's mental model is "I see the screen, I tap on a button, it should click there" — same as voice mode. The cursor position was completely disconnected from where the user tapped.

**Fix:** Changed trackpad taps to use absolute coordinates via `getNormalizedCoords(touch.clientX, touch.clientY)` — same coordinate math as voice mode taps. Sends `{ type: "tap", x, y }` with normalized 0.0-1.0 coords. Also syncs `pcCursorPos` to the tapped position so relative drags continue from the right spot.

**Key Rule:** Any tap-to-click gesture (voice or trackpad) MUST use `getNormalizedCoords()` with `videoRef.current.getBoundingClientRect()` — never `window.innerWidth`. The video uses `object-contain` which letterboxes, and may be zoomed via CSS transform. `getBoundingClientRect()` accounts for both.

### 2026-03-26 — Coordinate math for zoomed video taps

**Error:** Potential for taps to land wrong when video is CSS-zoomed.

**Root Cause:** `getBoundingClientRect()` returns the visual (scaled) rect of the video element. When zoomed 2x, the rect is 2x the viewport. `getNormalizedCoords` computes `(clientX - rect.left) / rect.width` which correctly maps to the visible portion of the video — e.g., tapping at center of screen while 2x zoomed on center = 0.5 normalized = center of PC screen.

**Fix:** No fix needed — `getNormalizedCoords` already handles zoom correctly because CSS transforms affect `getBoundingClientRect()`. This is why we must always use the video element rect, never `window.innerWidth`.
